프로젝트 파일

LifeSync AI는 4개 도메인(요리/운동/건강/취미)을 하나의 유기체로 통합 관리하는 AI 에이전트입니다.

핵심 가치: "한 가지 선택이 모든 영역에 연쇄 효과를 만든다"

예: '야식 라면' → 수면 품질 -35% → 다음 날 운동 성능 -20% → 체중 목표 +2일 지연

기술 스택

영역기술
BackendPython 3.11, FastAPI, WebSocket, LangGraph, LangChain
AI/MLGPT-4o-mini, OpenCLIP(ViT-B-32), MediaPipe, YOLOv8, PPO(Stable-Baselines3), Optuna
DatabaseChromaDB (Vector), Supabase PostgreSQL (State)
FrontendReact 18, Recharts, Tailwind CSS, Capacitor (Android APK)
InfraRailway ($5/월), Vercel (무료 CDN), AWS S3, Docker Compose

4. Orchestrator (LangGraph)

실행 흐름: classify_intent → route_to_agent → [food|exercise|health|hobby] → merge → cascade → evaluate

'치킨 먹었어' → food 분류 → FoodAgent → 칼로리 분석 → CASCADE: 수면-35%, 운동-20% → 게이지 즉시 반영 (v0.3.0 연결)

출발영향연쇄 효과
foodhealth영양 점수 x 0.3 → 건강 지표 변화
foodexercise칼로리 x 0.1 → 운동 에너지 변화
exercisehealth운동 강도 x 0.5 → 체력 변화
exercisemood운동 시간 x 0.2 → 기분 변화
hobbyhealth스트레스 감소 → 폭식 충동 -40%

5. 도메인 에이전트 (LangChain)

FoodAgent

GPT-4o-mini (temp=0.7) + ChromaDB 10만건 RAG → 3개 레시피 추천

ExerciseAgent

미세먼지/부상 고려 운동 추천. 실내 자동 전환.

HealthAgent

GPT-4o-mini (temp=0.3, 보수적) → 건강검진 3줄 요약 + 위험도

HobbyAgent

스트레스/기분 기반 취미 추천. 기타 30분 → 스트레스 -15%

6. 강화학습 엔진

LifeEnv: 40차원 상태, 10개 행동, 84스텝(12주)

보상: R = w1*taste + w2*health + w3*fitness + w4*mood + w5*habit - penalties + bonuses

v0.3.0 연결: 시뮬레이터에서 사용자 행동 시 PPO가 최적 행동을 함께 추천합니다.

행동보상연쇄 효과
야식 라면 (23시)-5수면 -35% → 운동 -20% → 체중 +2일
에어프라이어 수용+2칼로리 -40%
기타 연주 30분+2스트레스 -15% → 폭식 -40%
건강검진 이행+4불안감 감소 → 전체 안정

12. 팀 분업 — 코어 + 플러그인 아키텍처

설계 원칙

코어(core/)는 서비스의 기본 동작을 보장합니다. 각 담당자는 플러그인(plugins/)을 통해 기능을 고도화합니다. 플러그인이 미구현된 슬롯은 코어의 폴백(규칙 기반)이 자동으로 동작합니다.

코어 프레임워크

파일역할
core/interfaces.py6개 Protocol 정의 (DomainAgent, KnowledgeBase, RLAgent, ImageAnalyzer, VoiceProcessor, RewardFunction). 팀원이 구현해야 할 메서드를 명확히 정의.
core/fallbacks.py규칙 기반 기본 구현 (LLM/GPU 없이 동작). BasicFoodAgent, BasicExerciseAgent, BasicHealthAgent, BasicHobbyAgent, BasicRLAgent, BasicImageAnalyzer, BasicVoiceProcessor.
core/plugin_registry.py싱글톤 레지스트리. register()/get()/status() 메서드. 미등록 슬롯은 폴백 자동 반환.

17. 용어 사전

A. 프로그래밍 기초

API (Application Programming Interface)

정의: 프로그램들이 서로 대화하는 규약 역할: 프론트엔드와 백엔드 사이의 요청/응답 통로 사용 이유: 표준화된 형식으로 데이터를 주고받기 위함 예시: POST /api/query에 '치킨 먹었어' 전송 → AI가 '칼로리 800kcal' 응답

FastAPI

정의: Python 비동기 웹 프레임워크 역할: 모든 API 요청을 처리하는 중앙 서버 사용 이유: 비동기 처리, 자동 Swagger 문서, Pydantic 타입 검증 장점: Django 대비 2-3배 빠름 단점: 커뮤니티/플러그인 적음

Frontend (프론트엔드)

정의: 사용자가 보는 화면 (브라우저/앱) 역할: 11개 페이지의 UI 렌더링 + 사용자 입력 처리 사용 이유: React 18의 컴포넌트 기반으로 UI를 모듈화

Backend (백엔드)

정의: 사용자가 보지 못하는 서버 측 로직 역할: API 처리, DB 접근, AI 추론, 인증 사용 이유: FastAPI(Python)로 AI 라이브러리와 자연스럽게 통합

WebSocket

정의: 양방향 실시간 통신 프로토콜 역할: 대시보드 게이지 실시간 업데이트 사용 이유: HTTP 폴링 대비 지연 90% 감소 장점: 서버가 먼저 데이터를 보낼 수 있음

Docker

정의: 컨테이너 가상화 도구 역할: 동일 개발/배포 환경 보장 사용 이유: '내 PC에서는 되는데' 문제 해결 장점: 환경 일관성 단점: Docker Desktop 설치 필요

B. AI/ML 기술

GPT-4o-mini (LLM)

정의: OpenAI의 경량 대규모 언어 모델 역할: 자연어 이해/생성 (레시피 추천, 건강 요약, 취미 제안) 사용 이유: 한국어 우수, GPT-4의 1/10 비용 장점: 빠르고 저렴 단점: 할루시네이션(없는 정보 생성) 가능

LangChain

정의: LLM 애플리케이션 프레임워크 역할: 4개 도메인 Agent의 LLM 호출 관리 사용 이유: 프롬프트+출력파서+체이닝을 선언적으로 구성 장점: LLM 교체 용이 단점: 추상화 레이어 오버헤드

LangGraph

정의: LangChain 기반 상태 그래프 엔진 역할: Orchestrator의 8노드 DAG 실행 사용 이유: 복잡한 조건 분기를 선언적으로 정의 장점: 상태 자동 관리, 디버깅 용이 단점: 병렬 실행 미지원

RAG (Retrieval-Augmented Generation)

정의: 검색 증강 생성 — LLM에 외부 지식을 주입하는 기법 역할: ChromaDB에서 관련 문서를 검색 → LLM에 전달 → 정확한 응답 사용 이유: LLM만으로는 최신 레시피/운동 정보를 모름 장점: 정확도 향상, 할루시네이션 감소 단점: 벡터 DB 관리 필요

PPO (강화학습)

정의: Proximal Policy Optimization — 안정적 RL 알고리즘 역할: 40D 상태 → 10개 행동 중 최적 선택 사용 이유: 규칙 기반은 개인차 미반영, RL은 피드백으로 학습 장점: 지속적 개선 단점: 초기 데이터 부족, CPU 학습 느림

OpenCLIP

정의: 오픈소스 CLIP (이미지-텍스트 크로스 모달) 역할: 사진을 512차원 벡터로 변환 → 유사도 검색 사용 이유: 사진 한 장으로 건강 상태 추정 가능 장점: 무료, 이미지+텍스트 동시 처리

YOLO (You Only Look Once)

정의: 실시간 객체 감지 모델 역할: 식재료 자동 인식 (80+ 카테고리) 사용 이유: CPU에서도 100ms 추론 가능 장점: 빠르고 정확 단점: 한국 음식 특화 부족

MediaPipe

정의: Google의 인체 인식 AI 역할: 33개 랜드마크로 자세 분석, 운동 폼 교정 사용 이유: 무료, CPU 실시간, 높은 정확도 장점: 잘못된 자세 → 부상 방지

Optuna

정의: 하이퍼파라미터 자동 최적화 라이브러리 역할: PPO의 lr/gamma/batch 등을 자동 탐색 사용 이유: 수동 튜닝 대비 체계적 Bayesian 탐색 장점: 100 trial로 최적 조합 탐색

C. 인프라 & 보안

Railway

정의: 클라우드 배포 플랫폼 역할: 백엔드 24시간 운영 사용 이유: $5/월, GitHub 자동 배포, 설정 간편 단점: GPU 미지원, 콜드 스타트

Vercel

정의: 프론트엔드 CDN 배포 플랫폼 역할: React 앱 글로벌 배포 사용 이유: 완전 무료, 빠른 로딩 장점: 글로벌 CDN으로 어디서든 빠름

Capacitor

정의: 웹 → 네이티브 앱 변환 프레임워크 역할: Android APK 빌드 사용 이유: 하나의 코드베이스로 웹+앱 장점: 카메라/파일시스템 네이티브 API 접근 단점: iOS 별도 추가 필요

JWT (JSON Web Token)

정의: 상태 무관(stateless) 인증 토큰 역할: 사용자 인증 — 로그인 후 모든 요청에 토큰 첨부 사용 이유: 서버 세션 없이 인증 가능 장점: 서버 부담 적음 단점: 토큰 탈취 시 만료까지 유효

OAuth

정의: 제3자 인증 표준 (카카오 로그인) 역할: 자체 회원가입 없이 카카오 계정으로 로그인 사용 이유: 비밀번호 직접 관리 불필요 장점: 보안 강화, 사용자 편의

Rate Limiting

정의: API 호출 횟수 제한 역할: IP별 분당 60회 제한, 429 응답 사용 이유: DDoS 공격 방어

CORS

정의: Cross-Origin Resource Sharing (교차 출처 리소스 공유) 역할: 허용된 도메인만 API 접근 가능 사용 이유: 외부 사이트의 무단 API 호출 차단

Supabase

정의: PostgreSQL 클라우드 DB 서비스 역할: 사용자 40D 상태 벡터 영속 저장 사용 이유: 무료 500MB, REST API 자동 제공 장점: 앱 종료 후에도 상태 유지 단점: 연결 수 제한, timeout 발생 가능

ChromaDB

정의: 벡터 데이터베이스 (임베딩 저장/검색) 역할: 4개 RAG 컬렉션 (recipe, exercise, health, hobby) 사용 이유: 무료, persistent 저장, 한국어 multilingual 장점: 10만건에서 0.1초 검색 단점: 대규모(100만건+) 성능 저하

IndexedDB

정의: 브라우저 내장 데이터베이스 역할: 오프라인 모델/데이터 캐싱 사용 이유: 네트워크 없을 때도 앱 동작 장점: 큰 용량 저장 가능

Protocol (Python)

정의: 구조적 서브타이핑 (duck typing 공식화) 역할: 플러그인 인터페이스 계약 정의 사용 이유: 구현해야 할 메서드를 명확히 정의 장점: 런타임 타입 체크 가능 (runtime_checkable)
